# Backcheck — Mémoire Contextuelle

## Vue d'ensemble du projet
**Nom :** Backcheck
**But :** Application desktop pour vérifier/auditer les backups de projets audiovisuels (vidéo, musique, photo, etc.)
**Utilisateur :** Niels — usage pro + perso, Windows, NAS Synology + disques durs internes/externes
**Langue :** Bilingue FR/EN

## Besoins de l'utilisateur (issus des échanges)
- **Structure des projets :** Basée sur des dossiers, ex: `D:\20220713_ODESZA_ITS_ONLY\` avec sous-dossiers (FINALS, MUSIC, PROJET, SCREENS, SCRIPT...)
- **Taille des projets :** 1 Go à 5 To
- **Emplacements de backup :** NAS Synology (lecteur réseau mappé), disque dur interne, disque dur externe (branché à la demande)
- **Méthode de backup :** Copie manuelle, une seule copie de backup par projet
- **Pas de convention de nommage** pour les dossiers
- **Ce qu'on vérifie :** Présence des fichiers (manquants), intégrité (corruption/checksums), sync source vs backup, liens médias cassés, différences de date/timestamp
- **Modes de comparaison :** Comparer source ↔ backup (deux dossiers sélectionnés), OU scanner un backup isolé
- **Périmètre de scan :** Un seul projet OU tous les projets d'un coup
- **Format de rapport :** HTML + PDF (usage perso, pas besoin d'être super poli)
- **Exécution :** À la demande (pas planifié)
- **Pas d'alertes nécessaires**
- **Historique :** Oui, veut voir les résultats des scans passés
- **Application desktop** (pas web, pas CLI)
- **Pas de préférence technique** — on a choisi Electron + React + TypeScript + Vite
- **NAS :** Synology (accès via lettre réseau mappée)
- **Gros problème :** Ne sait pas si ses backups sont à jour
- **A déjà eu des fichiers corrompus** par le passé
- **Nom de l'app :** Backcheck

## Stack technique
- **Framework :** Electron + React 18 + TypeScript
- **Bundler :** Vite 6
- **Base de données :** sql.js (WASM, pur JS — pas besoin de compilation native)
- **Rapports :** Génération HTML (custom), PDF via `printToPDF` d'Electron
- **Thème :** Dark theme, CSS custom (pas de framework)

## Fonctionnalités clés
1. **Moteur de scan :** Parcours récursif des dossiers, comparaison de taille (rapide), checksum SHA256 (approfondi)
2. **Progression du scan en temps réel :** 4 phases (listing_source, listing_backup, comparing, checksums) avec barre de progression, compteur de fichiers et nom du fichier en cours
3. **Statuts des fichiers :** ok, missing_in_backup, missing_in_source, modified, error
4. **Synchronisation backup :** Bouton pour copier fichiers manquants/modifiés + supprimer orphelins, avec progression temps réel
5. **Historique SQLite :** Stocke chaque scan avec tous les résultats de fichiers
6. **Rapports HTML/PDF :** Cartes de résumé stylisées + table détaillée des fichiers
7. **Barre de filtres :** Filtrer les résultats par statut (ok, manquants, modifiés, erreur)
8. **i18n :** FR/EN complet, persisté dans localStorage
9. **Page d'aide :** 9 sections détaillées expliquant le fonctionnement de l'app

## Bugs corrigés
- `better-sqlite3` → remplacé par `sql.js` (WASM, pas de compilation native)
- `ELECTRON_RUN_AS_NODE=1` était défini dans l'environnement système → forçait Electron à se comporter comme Node.js
  - **Solution :** Lancer via PowerShell avec `$env:ELECTRON_RUN_AS_NODE=''`
  - Script `npm start` mis à jour avec `set ELECTRON_RUN_AS_NODE=`
- Clé i18n `scan_files` dupliquée (History vs Scan) → renommée `scan_progress_files`

## Commandes pour lancer l'app
```powershell
# PowerShell (dev)
$env:ELECTRON_RUN_AS_NODE=''; & 'node_modules\electron\dist\electron.exe' .

# Ou via npm (cmd)
npm start

# Ou lancer le .exe portable
.\release\win-unpacked\Backcheck.exe
```

## Problèmes connus restants
- Les warnings GPU cache ("Unable to move the cache") sont normaux et inoffensifs
- Le fichier WASM de sql.js doit être accessible à l'exécution — peut nécessiter une config supplémentaire
- Le preload utilise `require()` (CommonJS) — fonctionne correctement

## Notes importantes pour les futures sessions
- **Clé i18n `scan_files`** = "Fichiers" dans la section History. Ne pas la dupliquer.
- **Clé i18n `scan_progress_files`** = "fichiers" pour la progression du scan.
- **Pattern IPC progression :** `onXxxProgress` / `offXxxProgress` dans preload, `xxx:progress` channel dans main, callback dans scanner.
- **Sync :** Le bouton n'apparaît que si le scan détecte des fichiers manquants/modifiés. Confirmation obligatoire avant action.
